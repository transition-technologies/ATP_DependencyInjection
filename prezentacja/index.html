Standardowa budowa aplikacji

  • obiekty same tworzą inne obiekty które są im potrzebne do wykonywania swoich zadań;
  • obiekty wiedzą wszystko (implementacja, ilość obiektów, wynik operacji) na temat używanych obiektów.
public class LongDecimalCalculator {
	private final LongDecimalNumeralSystem system;
	
	public LongDecimalCalculator() {
		system = new LongDecimalNumeralSystem();
	}
	
	public String add(final String first, final String second) {
		return system.add(first, second);
	}

Standardowa budowa aplikacji

Zalety:

  • łatwo odnaleźć się w kodzie oraz w przebiegu procesu działania aplikacji.

Wady:

  • duża zależność między obiektami;
  • utrudnione wprowadzanie zmian w działaniu aplikacji (tworzenie obiektów konkretnych klas w wielu miejscach);
  • utrudnione testowanie aplikacji (brak możliwości podmiany używanej implementacji).

Inversion of control

Techniki:

  1. wstrzykiwanie zależności;
  2. mechanizm zdarzeń;
  3. programowanie aspektowe.
Każdy kolejny poziom stwarza większe możliwości rozbudowy aplikacji ale jednocześnie powoduje większą "utratę" kontroli nad jej działaniem.

Inversion Of Control

Zalety:

  • rozluźnienie powiązań między obiektami;
  • możliwość podziału aplikacji na niezależne moduły;
  • ułatwione wprowadzanie zmian oraz rozbudowa aplikacji;
  • ułatwiony proces testowania automatycznego.

Wady:

  • utrudnione odnajdywanie przebiegu procesu działania (co, kiedy, dlaczego i jak się wykona?).

Dependency Injection

  • obiekty dostają swoją konfigurację podczas ich tworzenia (konstruktory, metody zwane setter'ami);
  • obiekty nie wiedzą jakiej implementacji klasy używają.
public class Calculator implements Calculate {
	private final NumeralSystem system;
	
	public Calculator(final NumeralSystem system) {
		this.system = system;
	}
	
	@Override
	public String add(final String first, final String second) {
		return system.add(first, second);
	}

Dependency Injection

Zalety:

  • ułatwione testowanie jednostkowe (różna konfiguracja obiektu do testów a inna na produkcji);
  • ułatwione testowanie integracyjne (możliwość zastępowania obiektów zaślepkami (mock'ami) w celu pominięcia ich wpływu na działanie testowanego obiektu).

Wady:

  • tworzenie obiektu wymaga najpierw stworzenia obiektów, z których będzie on korzystał.
